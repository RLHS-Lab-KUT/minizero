### 1.2 n スイープ（取得・分析済み）

決定論 eval config で n = 16 / 24 / 32 / 64 / 128 / 256 の記録を取得済み．

- **halving 発火閾値は n=32（実測確認）**．n=16/24 では発火しない．
- 方策順位の逆転：n=16 は 0/55 ply，n=24 は 0/58 ply，**n≥32 は 42–43/59 ply**．
- **n を増やしても強さは向上しない**（評価差 ≤ 0.17）．
  → 研究上の n の価値は「強さ」ではなく「説明可能性」にある．

### 1.3 Gumbel パラメータ（ソース確認済み）

- `actor_gumbel_sample_size`（= m）：既定値 **16 の固定値**．論文 Go 実験の m=min(n,16) ではない．
  [configuration.cpp L31]
- 初期 budget：`max(1, floor( n / (log2(m) * sample_size_) ))` [gumbel_zero.cpp L99]
- **発火判定に使うのは next_budget**：`floor( n / (log2(m) * sample_size_ / 2) )`，
  条件は `next_budget > 0 && sample_size_ > 2` [gumbel_zero.cpp L109-110]
  → 過去メモの `floor(16/32)` は **正しい**（分母は sample_size_/2 側）．
- halving は `sample_size_ > 2` で停止する [L110]．

### 1.4 tree_json の拡張フィールド

### 2.3 上位2手の訪問数タイ [事実]（2026-07-21 実測で確定）

当初の仮説「n をいくら増やしても上位2手の訪問数が同数になる」は，
そのままの形では成立しない．実測は n 依存だった．

| n | 生存2手の ply | N 一致 | 一致率 |
|---|---|---|---|
| 16 | 1/64 | — | 検証対象外（halving が1巡も走らない） |
| 32 | 58/64 | 38 | 65.5% |
| 128 | 56/64 | 56 | 100% |

n=32 の不一致 20 件は**すべて差が 1**（7 対 6，8 対 7 等）．
[推論] n=128 では予算式の各段が割り切れる（2→6→14→30）が，
n=32 では floor で端数が落ちるため，最後の配分で 1 回ぶんの余りが出ると考えられる．
[要確認] 未検証．

**説明設計にとっての帰結は変わらない** [推論]：
問うべきは「訪問数の差が手の優劣を語る根拠になるか」である．
n=128 では差ゼロ，n=32 でも最大 1（7 対 6）であり，
「エージェントはこの手を最も多く読んだ」という説明はどの n でも成立しない．
根拠が「構造上必ず同数」から「実測すると差は 0 か 1」に変わっただけで，
説明の語り口を訪問数以外に置く必要がある点は変わらない．
